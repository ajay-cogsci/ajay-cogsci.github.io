Design Brief · Governance + R&D · escape-testing for frontier models

Keep the boundary. Drop the second device.

Your Artifact 3 is strong but over-scoped for the budget. This brief shows how to collapse it to a single-host sandbox that still holds against major frontier models, which benchmarks validate it, how to iterate it, and how to beat CyBench saturation without burning tokens — grounded in what the AISIs, METR, CAISI and the frontier labs actually run.

Executive summary — five decisions
  • Single host is defensible. The dedicated control device becomes host-level processes; the microVM boundary is what makes them "out-of-band". This is the pattern SandboxEscapeBench already uses on one machine. [1]
  • Validate holding separately from capability. Use SandboxEscapeBench to prove the sandbox holds; use CyBench / ExploitBench to prove the agent is capable enough for holding to matter. [1,2,8]
  • You are not alone at CyBench saturation. CAISI and the AISI network hit the same ceiling; the field's move is graded metrics + harder sets, not more epochs. [3,4]
  • Tokens are the cost lever. An optimised scaffold reaches the same score at ~13% of the token budget; a 100M-token cyber run costs ~$80. Spend smarter, not harder. [5,6]
  • Residual risk, not zero risk. Collapsing the air-gap to a logical boundary is acceptable under a stated residual-risk threshold with senior sign-off — the AISI protocol model. [11]

04

Which benchmarks validate the sandbox

Two separate questions. Does the box hold? is answered by escape benchmarks run against your own sandbox. Does holding matter? is answered by capability benchmarks that tell you how hard the agent is pushing. Run both; report them separately.

A · DOES THE BOX HOLD?

SandboxEscapeBench

Primary validator

The validation set for a sandbox. 18 scenarios across orchestration (L1), runtime (L3), kernel (L4). Run it against your own outer boundary: the agent tries to escape the container into the host, where the flag sits. Binary, host-side scoring means no reward hacking. [1]

R&D detail

Run the solution variant first to confirm your harness works and each scenario is reachable, then the adversarial variant to confirm your boundary is not bypassable by an unintended path. SandboxEscapeBench found 4 shortcuts in development (default SSH creds, mismatched-CVE eBPF) that had to be hardened so escapes used the intended exploit only.

inspect eval sandbox_escape_bench \
  --sandbox vagrant     # VM-in-host provider = single machine
  -T variant=adversarial --epochs 5 --token-limit 2500000

Escape success scales log-linearly with compute; run difficulty-tiered so you see which scenarios flip fail→pass as models improve. [1]

Harness & shortcut audit

Recommended

The harness is itself an attack surface. Confirm the host flag is reachable only via a real escape, that no info leaks into the container, and that the scorer can't be gamed. Audit transcripts of every "success" to verify the intended attack path. [1]

R&D detail

Automate transcript review (Meerkat / BenchJack-style auditors) rather than manual spot-checks. Seal any git history / package caches the agent could read a solution from; treat a "solve" with an unexpected trajectory as a harness bug, not a capability signal.

B · IS THE AGENT CAPABLE ENOUGH TO MATTER?

CyBench

Baseline · saturating

40 professional CTF tasks; the community baseline (o1 hit ~45% in the CAISI/UK joint test, and newer models are higher). Keep it for comparability and its subtask decomposition — but expect saturation at the frontier. [2,3]

R&D detail

Report pass@1 at a fixed budget for comparability with the network's runs (Inspect, ~2.5M token cap, temp 0.7–1). Use subtask completion as the gradient signal, not just full solves — see §06.

ExploitBench

Adopt · beats saturation

Capability-ladder benchmark: 16 deterministically-graded flags across 5 tiers, up through escaping the engine sandbox to arbitrary code execution, on 41 hardened V8 bugs. Gives resolution between models that both "pass" CyBench. Publicly deployed frontier models routinely crash but rarely reach ACE. [8]

R&D detail

The ladder tiers — reach → trigger → in-cage primitive → escape the engine → control — mean you score "how far up did it climb", not "did it pop". That is exactly the anti-saturation metric you need, and the escape tier is directly relevant to your threat model. Deterministic oracles (randomised challenge-response, differential execution) avoid LLM-judge fragility.

InterCode-CTF + NYU CTF Bench

Port to Inspect

InterCode is already part of the AISI-network agentic-cyber setup alongside CyBench; NYU CTF Bench adds 200 tasks. Both broaden coverage and are worth the porting effort you're already doing. [4,5-refs]

R&D detail

The network ran CyBench + InterCode on Inspect at temp 0.7, 10 samples/task, 2.5M-token caps — mirror those settings for cross-org comparability.

CVE-Bench / CyberGym · CyberSecEval-PI

Optional realism

Real-world exploitation (CVE-Bench ~10–13%; CyberGym 1,507 vulns) tests whether capability generalises off synthetic CTFs. CyberSecEval's prompt-injection set tests the trigger: does an injected instruction become a tool-abuse / escape attempt? [ref]

R&D detail

These are heavier to stand up. Treat as periodic realism checks, not per-release gates, unless budget allows.

05

How to iterate the sandbox

gates = governance sign-off points

A phased path from your current CyBench-on-Inspect setup to a validated, continuously-tracked escape programme. Each phase ends in a gate — a decision the R&D team hands to governance.

PHASE 0 · you are here

Single-host nested sandbox on Inspect

Stand up Firecracker + gVisor + container on one host with the VM sandbox provider; keep CyBench running; finish porting InterCode / NYU CTF into Inspect.

Gate: harness reproducible, logs captured out-of-band.
PHASE 1

Prove it holds

Add SandboxEscapeBench via the VM provider. Run the solution variant (harness sanity) → adversarial variant with a frontier model → harden every unintended escape path until success only follows the intended exploit.

Gate: no unintended escape paths; residual-risk statement drafted.
PHASE 2

Red-team the harness

Auto-audit transcripts of all successes; seal information leakage and reward-hacking routes; confirm the host flag is reachable only via a real breakout.

Gate: scorer integrity signed off; first validated escape-capability number published internally.
PHASE 3

Continuous tracking

Re-run the escape suite on every new frontier model; watch for scenarios flipping fail→pass; maintain a private held-out set; track against the ~7-month autonomy-doubling trend. [1,ref]

Gate: if-then thresholds defined — which escape result triggers which containment upgrade.
PHASE 4 · as budget allows

Deepen coverage & add control

Extend to app-layer (L2) and novel-vuln discovery; layer a lightweight monitor / control protocol (ControlArena-style) once capability rises enough that "does monitoring catch it?" becomes the live question.

Gate: escape + control results reviewed together at each release.
06

Beating CyBench saturation without burning tokens

The trap: to separate two models that both "pass" CyBench, the obvious move is more epochs and bigger budgets — but AISI shows cyber success keeps rising log-linearly to 100M tokens with no plateau, and a 100M-token run costs ~$80. Brute force is expensive and doesn't cleanly rank saturated models. The fix is to measure differently and spend smarter. [5,6]

Lever 1

Optimise the scaffold first

AISI reached the same 25% cyber dev-set score using ~13% of the token budget of a non-optimised scaffold — just a better system prompt and tool access. This is the single biggest cost lever. [5]

free
Lever 2

Score subtasks, not solves

CyBench ships subtask decompositions. Grade partial progress to separate saturated models without a single extra token. [2]

free
Lever 3

Switch the axis to token-efficiency

At a fixed modest budget, measure tokens-to-solve / progress-per-token — AISI's "token efficiency" dimension. Differentiates strong models cheaply instead of chasing asymptotic pass rates. [6]

lower total
Lever 4

Adaptive budgeting + early stop

Fixed cap (e.g. 2.5M tokens), pass@1 or low epochs; stop early on trivial solves and on stalls/loops. AISI found models were usually still progressing at cutoff — so terminate when they are not. Use input caching (note: context compaction resets it). [4,7]

fewer runs
Lever 5

Prune to the discriminating tail

Drop tasks every frontier model solves trivially (no signal) and every task none can touch. Keep the hard, contamination-safe middle + recent/private tasks. Fewer tasks × lower epochs = far fewer tokens.

headroom
Lever 6

Move up-difficulty, not up-epochs

Add ExploitBench and cyber-range-style multi-step tasks where there is still headroom, rather than running CyBench at ever-higher k. Report pass@1 with a stated budget, not pass@k with large k. [8,ref]

Governance framing

Report escape-capability as "success at budget X" with X fixed and stated, plus a ladder/subtask breakdown — never an uncapped pass@k, which inflates with spend and is not comparable across releases. This keeps the number honest, cheap, and defensible.
