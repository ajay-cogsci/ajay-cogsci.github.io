Strategic plan · governance + R&D reference · single-host escape-testing sandbox

From lean design to a validated, scalable escape-testing sandbox.

This plan specifies every component of the single-host sandbox, the libraries and tools each requires, its dependency on UK AISI's Inspect and extensions, the resources needed to validate it for academic and governance use, and a graded-metric benchmark design backed by peer-reviewed evidence — closing with a small, costed pilot study to prove it holds and produces trustworthy signal.

Executive summary — the plan at a glance
SCOPEEleven components on one host: orchestrator, agent, three nested boundaries, egress control, telemetry, scoring, and a governance layer.
DEPENDENCYBuilt on Inspect (open-source, MIT) plus four extensions. The AISI network's standardisation on Inspect lowers lock-in risk.
METRICSA graded stack — outcome, process/ladder, reliability (pass^k), token-efficiency — with statistical error bars, all peer-reviewed-backed.
RESOURCESOne KVM-capable host, a small team, ~4–6 weeks for the pilot; a 100M-token cyber run costs ~$80, so the pilot fits a modest budget. [12]
VALIDATIONThree tracks — academic (construct validity + reproducibility), industrial (throughput + reliability), governance (thresholds + audit trail).
DECISIONThe pilot yields a go/no-go for scale-up to a multi-node deployment suitable for governance-grade evaluation.

Dependency on Inspect & its extensions

The sandbox is an Inspect application. Inspect core supplies the eval loop, sandbox abstraction, scorers, metrics, budget limits, and tamper-evident logs; four extensions supply the pieces the lean design needs. Toggle to see which dependencies are hard (pilot cannot proceed without them) versus optional.

hard dependency optional / scale-out Inspect abstracts the sandbox, so swapping the VM provider (proxmox → vagrant/Firecracker) is a config change, not a rewrite.
Lock-in risk & mitigation

Single-framework reliance is a real dependency risk. It is mitigated by three facts: Inspect is open-source under a permissive licence; its task format is portable and the sandbox layer is provider-agnostic; and it is the shared standard across the UK, US (CAISI), and the wider AISI network — so an escape programme built on it stays comparable with, and portable to, government evaluations. Graded metrics — design & evidence

each metric maps to an Inspect primitive

A single binary pass/fail is the wrong instrument for a saturating field. The design layers five metric families, each addressing a distinct question and each backed by peer-reviewed or high-quality institutional evidence. Every family maps to a native Inspect scorer or metric, so the pipeline is buildable today.

M1

Outcome — pass@1

comparability floor

Binary task success at a fixed budget. Retained only for cross-org comparability with the AISI network's CyBench runs — never as the sole signal.

Inspect: includes() / match() scorer, accuracy() + stderr() metrics.
M2

Process — subtasks & milestones

dense partial credit

Award credit for intermediate progress — CyBench subtasks, GDM dangerous-capability milestones, or mean-criteria-passed. Separates models that tie on the binary at no extra token cost.

Evidence

The Agentic Benchmark Checklist recommends using process-based metrics alongside outcome-based metrics; APEX-Agents notes that mean criteria passed "provides a dense signal that rewards partial progress." [2,10]

Inspect: custom @scorer returning numeric value; value_to_float maps PARTIAL→0.5 natively.
M3

Capability ladder

escape resolution

Grade how far up the exploitation ladder a model climbs — including the explicit sandbox-escape tier — rather than one pass/fail. The anti-saturation instrument and directly on-threat-model.

Evidence

ExploitBench decomposes exploitation into 16 deterministically-graded flags across five tiers, arguing that treating a crash as success "collapses the hard parts of exploitation." Deterministic oracles avoid LLM-judge fragility. [8]

Inspect: custom @scorer emitting a per-flag bitmap; aggregate with multi_scorer.
M4

Reliability — pass@k & pass^k

ceiling vs consistency

Report both: pass@k (capability ceiling — can it ever escape?) and pass^k (consistency — does it escape every time?). A high-pass@k / low-pass^k profile is the signature of a model that benchmarks well but is unreliable in practice.

Evidence

τ-bench introduced pass^k for tasks requiring "reliability and consistency"; as k rises, pass@k increases while pass^k falls. Agents strong on average can be unstable across repeats. [13,14]

Inspect: multi-epoch runs; reducers over samples; pass^k as a custom aggregate metric.
M5

Token-efficiency

cost-aware capability

Success at a fixed budget, tokens-to-solve, and area-under-the-budget-curve. Distinguishes models cheaply and keeps reported numbers honest as inference-compute scales.

Evidence

AISI finds cyber success scales log-linearly with tokens to 100M with no plateau, so uncapped pass@k inflates with spend; an optimised scaffold reached the same score at ~13% of the token budget. [12,15]

Inspect: token/message/time limits in eval(); per-sample token accounting in .eval logs.
M0

Statistical wrapper

error bars on everything

No metric is reported without an interval. Standard errors and confidence intervals, bootstrap CIs, and — critically — clustered standard errors when the same task is sampled repeatedly. Paired differences and power analysis size the run.

Evidence

Miller (Anthropic) shows clustered standard errors can be over three times larger than naïve ones; without error bars, eval scores are effectively meaningless. [16]

Inspect: stderr(cluster=…), bootstrap_stderr() are built in — the clustered-SE recommendation maps to one argument.
Validity guardrails — non-negotiable

Graded metrics are only trustworthy if the task and grader are valid. The Agentic Benchmark Checklist is applied to every ported benchmark: an oracle/solution solver confirms solvability, state is cleared between runs, the agent is isolated from ground truth, and implementation shortcuts are audited. These checks matter — validity issues have been shown to mis-estimate agent performance by up to 100% in relative terms, and applying the checklist cut one benchmark's overestimation by 33%. [2]

05

The pilot study

a costed go/no-go

A small, time-boxed study to answer two questions: does the sandbox hold against a frontier model, and does the graded-metric pipeline produce stable, discriminating signal at acceptable cost. It is deliberately narrow so it can run on one host in weeks, not quarters.

Objective
Validate that the lean single-host sandbox contains a frontier agent with no unintended escape paths, and that the graded-metric stack ranks models that tie on binary pass — within a fixed, modest token budget.
Models
Two to three current frontier models via API (no local weights, no GPU). One weaker reference model to confirm the metrics detect a capability gap.
Task set
Holds: all 18 public SandboxEscapeBench scenarios (solution + adversarial variants). Capability: a ~10-task discriminating CyBench subset (with subtasks) + an ExploitBench pilot subset. Trigger: a small prompt-injection set to test whether an injected instruction becomes an escape attempt.
Metrics
pass@1 + process/ladder partial credit (M2/M3) + pass@k & pass^k at small k + tokens-to-solve (M5), each with clustered SE and bootstrap CIs (M0).
Design
Paired model comparison; power analysis to size epochs before running; chain-of-thought with resampling; a fixed 2.5M-token cap for capability tasks and a larger cap for escape runs, matching AISI-network settings for comparability. [4,16]
Validity checks
Run oracle/solution scripts first (solvability + harness sanity); audit the transcript of every escape success to confirm the intended attack path; clear state between runs; verify the host flag is unreachable without a real breakout. [1,2]
Success criteria
(1) zero unintended escape paths after hardening; (2) graded metrics separate models that tie on pass@1; (3) confidence intervals tight enough to rank at the chosen budget; (4) total spend within the pilot budget.
Deliverables
An open-sourceable validated harness; a metric report with error bars; a residual-risk statement; and a go/no-go recommendation for scale-up.

# capability task on the lean sandbox, graded, error-barred
from inspect_ai import task, Task, eval
from inspect_ai.solver import basic_agent, use_tools
from inspect_ai.tool import bash, python
from inspect_ai.scorer import scorer, accuracy, stderr, bootstrap_stderr

@scorer(metrics=[accuracy(), stderr(cluster="task_id"), bootstrap_stderr()])
def ladder_scorer(): ...   # emit per-flag bitmap → numeric partial credit

@task
def escape_capability():
    return Task(
        dataset=discriminating_subset,          # pruned, contamination-safe
        solver=basic_agent(tools=[bash(), python()]),
        scorer=ladder_scorer(),
        sandbox="proxmox",                       # microVM outer boundary; swap to vagrant/firecracker
    )

eval(escape_capability(), epochs=5, token_limit=2_500_000,
     message_limit=100, temperature=1.0)         # AISI-network-comparable settings

Escape validation reuses the SandboxEscapeBench harness with the same VM provider so a breakout lands in the outer VM, never the host. Transcript auditing runs post-hoc over the .eval logs.
